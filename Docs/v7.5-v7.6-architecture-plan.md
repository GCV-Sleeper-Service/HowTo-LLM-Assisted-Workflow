## 1. Current Project Assessment

### 1.1 What the project is today

This is an ESP32-C3 SuperMini running ESPHome on ESP-IDF (not Arduino), scanning ThermoPro TP357 BLE sensors and serving a fully self-contained HTML dashboard directly from flash. No Home Assistant, no database, no cloud dependency required for core value.

The current baseline (v7.4.5.1) supports 1–4 ThermoPro sensors with:

- 24 hours of 15-minute averaged history in RAM ring buffers
- Up to 45 days of hourly history persisted to a dedicated 512 KiB NVS partition
- An embedded dashboard with Chart.js charting, dark/light theme, CSV export, CSV import (both destructive multi-sensor and non-destructive single-sensor merge), custom date range selection, dew point calculation, comfort estimates, and min/max tracking
- A manifest-driven configuration workflow (`config/sensors.json` → Python generator → C++ header + YAML + JS + test fixtures)
- Playwright browser regression tests with fixture-driven mock server
- CI pipeline, preflight validation, and comprehensive documentation

### 1.2 Why this project stands out

Most ESP32 sensor projects stop at "serial log + raw MQTT" or "send readings to Home Assistant." This one is materially different in several ways.

**It is a self-contained product, not a component.** The gateway serves its own dashboard, persists its own history, handles its own export/import — it is usable by opening a browser to the device IP, period. That is a rare trait in the ESP32 ecosystem and it is the core of the business value proposition.

**The engineering discipline is unusually high.** The repo has a changelog with 32+ bug entries and 38+ operational lessons. It has Playwright browser tests (28/28 passing across fixture variants). It has a canonical manifest workflow that eliminates four-file manual configuration drift. It has preflight validation with ~12 sensor-count checks. It has a documented fresh-start handoff. These are practices you typically see in commercial embedded products, not hobby builds.

**The build pipeline is thoughtful.** The `generate-header.sh` → minification → C++ header embedding flow is the right approach for this scale. The Python generator reads a single manifest and produces all dependent artifacts. The pipeline avoids runtime JSON parsing on the ESP32 entirely — everything is resolved at compile time.

**The import/export system is surprisingly robust.** The transport went through four design iterations (POST body → query params → custom headers → URL path encoding) to work reliably through Cloudflare tunnels. The single-sensor merge import builds an epoch-to-slot map and overlays data non-destructively. That level of iteration on a specific problem is a sign of real product thinking.

### 1.3 Current limitations

The project is architecturally optimized for one thing: ThermoPro temperature/humidity monitoring. That is both its strength and its constraint.

**The C++ data model is ThermoPro-shaped.** `SensorSlot` has hardcoded `temp_sum`, `hum_sum`, `temp_history`, `hum_history`, `add_temp()`, `add_hum()`, `compute_and_format()`. The persistence format (`SegmentSnapshot`) stores `temp[NUM_SENSORS][POINTS]` and `hum[NUM_SENSORS][POINTS]` arrays. Adding a light sensor or a ping probe means either shoehorning data into temp/hum fields or rewriting the storage engine.

**The dashboard is semantically climate-bound.** `buildSensorCards()` generates HTML with hardcoded temperature, humidity, dew point, and comfort sections. `fetchSensorHistoryRows()` fetches exactly `/history/{id}/temp` and `/history/{id}/hum`. The export functions build CSV columns around `temp_c`, `temp_f`, `humidity_pct`, `dewpoint_c`. A new sensor type would require touching many functions.

**The manifest is identity-only.** Current `sensors.json` (schema v1) contains `{id, name, mac}` — enough for ThermoPro count control, but it carries no information about what kinds of data a sensor provides, what units it uses, or how to render it.

**Aggregation is conceptual.** The API endpoints exist (`/api/status`, `/api/storage-stats`) that could feed a multi-gateway view, but there is no discovery contract, no multi-gateway dashboard mode, and no defined satellite-to-aggregator protocol.

**The C3 is the right satellite, but not the right aggregator.** The ESP32-C3 has ~320 KB of usable RAM and a single core. It is an excellent choice for a satellite node, but trying to make it poll multiple gateways, cache their manifests, and serve a unified dashboard would push it past its resource limits.

### 1.4 Version label clarification

An important note about version numbering: the current `future-plans.md` has shifted the feature labels since the earlier plans. In the current roadmap:

- v7.5.x → Secrets/settings persistence review
- v7.6.x → Encrypted secrets partition
- Modular Architecture / Gateway Aggregation → listed separately as a major future effort
- Dashboard Pane Extensibility → listed separately

Throughout this document, "v7.5" and "v7.6" refer to the **feature concepts** (gateway aggregation and sensor extensibility), not the exact version numbers. The actual release labels should be decided when work begins, based on what has shipped by then.

## 2. Understanding v7.5 and v7.6 as Defined in the Repo

### 2.1 v7.5 — Modular Architecture / Gateway Aggregation

The repo's `future-plans.md` describes this as enabling "multiple BLE gateways (e.g., one per building) to have their data aggregated by a central gateway's dashboard."

It identifies three possible approaches (pull model, push model, iframe/embed) and recommends starting with the pull model because "it requires no changes to the satellite gateways (they already expose everything needed)."

The key insight in the repo's own assessment is that the design effort should happen now but the implementation should be deferred until after the foundation features are solid. The existing `/api/status` endpoint is explicitly called out as the starting point for the data contract.

### 2.2 v7.6 — Dashboard Pane Extensibility / Sensor Extensibility

The repo's `future-plans.md` contains the critical sentence: "This requires abstracting the current ThermoPro-specific `SensorSlot` into a more generic sensor model with type-specific rendering. It's a substantial refactor of both the C++ backend and the JavaScript dashboard."

It also contains the most important piece of wisdom in the entire roadmap: "Design the abstraction when working on 7.5 (modular architecture), but implement it only when there's a concrete second sensor type to support. Don't build a generic framework speculatively."

That caution is exactly right. The abstraction needs to be designed now because aggregation depends on it (the aggregator needs to know how to discover and render arbitrary sensor types). But the full backend implementation should be validated with a real second sensor family, not built speculatively.

### 2.3 Why these two features are coupled

Aggregation and sensor extensibility are architecturally coupled because the aggregator's job is to ask satellites: "what do you have?" and then render it. If the answer is always "temp and hum," you do not need a generic contract. But the moment you want the aggregator to handle a satellite with a ping probe or a host health monitor, you need a manifest that describes sensor types, metric definitions, and rendering hints.

The manifest v2 is the bridge between these two features. It serves both purposes:

- The aggregator uses it to discover what sensors a satellite has and how to render them
- The local dashboard uses it to render non-climate sensors alongside climate sensors

This is why the manifest design should come first, before either aggregation or sensor extensibility is fully implemented.

## 3. Feedback Source Evaluation

### 3.1 Source GE (files: feedback-GE.md, feedback-GE-update.md)

**Strong points:**

- Correctly identifies the shift from "Data-Aware" to "Metadata-Aware" architecture as the core conceptual change. That framing is accurate and useful.
- The compile-time generation approach (Python generator produces static C++ structs from the manifest) is the right call for ESP32. The "Bounded Array Trick" with `MAX_METRICS_PER_SENSOR` is a practical memory safety pattern.
- The phased rollout sequence (manifest schema → dashboard → firmware → aggregation) is sensible.
- The recognition that "Single Codebase, Configurable Roles" is a good design principle — one firmware binary, role determined by config.
- Good concrete example of the new manifest v2 schema with gateway metadata and per-sensor metrics arrays.

**Weak points:**

- **The initial GE draft says aggregation should be browser-driven pull (browser fetches multiple satellites with CORS `*`).** This is the weakest recommendation across all feedback sources. It works in a single-VLAN home lab but breaks in any real deployment with VLANs, VPNs, corporate firewalls, or security requirements. Setting `Access-Control-Allow-Origin: *` on every satellite is an operational and security burden.
- **The GE update then contradicts this** by proposing on-device aggregator polling `GET <satellite>/api/latest` every 60 seconds. This is a better direction but the two documents together leave a mixed message.
- **Proposes dynamically sized NVS blobs based on total metric count in the manifest.** This is risky. Coupling manifest edits to mandatory history backup → wipe → migrate → restore is a poor user experience and a source of data loss bugs. The current fixed-size segment layout is a strength, not a weakness.
- **Suggests per-sensor compile-time structs (`SensorSlot_rack_1`)** which is overly specific and harder to maintain than a bounded generic array.
- **Missing:** No discussion of backward compatibility shims, no API versioning strategy, no testing strategy, no history storage migration plan beyond "wipe and restore."

### 3.2 Source PE (file: feedback-PE-Unified.md)

**Strong points:**

- **The most operationally grounded of the three.** It explicitly addresses backward compatibility, testing considerations, migration paths, and the "why not" for each architectural alternative.
- **History storage strategy is the most conservative and correct.** Keep flash format unchanged for environmental metrics; use RAM-only buffers for new categories initially; defer generalized flash format to v8.x when patterns stabilize. This avoids the highest-risk refactor.
- **Aggregator-as-device (not just browser-driven pull) with clear rationale.** The argument against browser-driven aggregation (VLAN/VPN issues, CORS burden, security surface) is well-reasoned and matches real deployment realities.
- **Recommends ESP32-S3 for aggregator** rather than trying to shoehorn aggregation into the C3. This is correct.
- **The comparative assessment section (Section 9)** is uniquely valuable — it evaluates GE's browser-driven CORS approach, fully generic charting, aggressive NVS resizing, and heavy STL usage, and explains why each is rejected. That kind of explicit trade-off reasoning is what makes a plan implementable.
- **The C++ model proposal** (MetricDef + MetricHistory + SensorEntity with fixed arrays and `const char*`) is clean, memory-safe, and maps well to the existing generator pattern.
- **Testing considerations are explicit** — extend Playwright tests per category, validate manifest-driven rendering.

**Weak points:**

- **Tends toward more complexity than may be needed in the first iteration.** For example, defining `MetricType` as an enum with 12 variants before having a second real sensor type is arguably premature.
- **The aggregator architecture section could be more explicit about failure modes** — what happens when a satellite is unreachable, how stale data is shown, how the aggregator handles a satellite whose manifest has changed.
- **Missing:** Explicit data model for the aggregator's in-memory cache, sizing estimates for aggregator RAM usage per satellite.

### 3.3 Source GP (files: feedback-GP.md, feedback-GP-Research.md)

**Strong points:**

- **The most architecturally mature of the three sources.** The "measurement-catalog thinking" framing and the three-layer split (Source Adapter → Logical Device → Measurements) is the cleanest conceptual model.
- **The measurement class taxonomy is excellent.** Five classes (analog numeric, binary state, discrete enum, counter/event, metadata-only) cover the full range of sensor types without over-generalizing. This is the right level of abstraction.
- **The manifest v2 design is the most detailed and most deployment-ready.** It includes `history_class`, `display` hints per measurement, `device_type` strings, and adapter-level source binding. It is specific enough to implement directly.
- **The API versioning strategy is the most practical.** Adding `/api/v2/gateway`, `/api/v2/catalog`, `/api/v2/live`, `/api/v2/history/{device}/{measurement}` alongside existing endpoints is clean, backward-compatible, and stable.
- **The "What Not To Do" section is critically important.** "Do not build a universal framework with no second real use case." "Do not make every measurement persist by default." "Do not break current endpoints in place." "Do not force the first aggregator into the current C3 firmware." Every one of these is correct.
- **Phase 5 (add one non-climate sensor family) with the recommendation to start with ping/HTTP health probe is a strong tactical choice** — no new hardware required, clear business value, forces the architecture to handle non-climate semantics.
- **The GP Research file** adds timeline diagrams (Gantt, ER diagram, architecture flow) and identifies gaps across all feedback documents (auth/trust boundaries, schema migration, config simplicity).

**Weak points:**

- **The research file's timeline estimates are optimistic.** Manifest v2 definition in 14 days, SensorEntity refactor in 21 days — these assume smooth implementation without the typical firmware debugging cycles.
- **The GP plan is the longest and most detailed, which is a double-edged sword.** An implementor could feel overwhelmed by the scope. The priority hierarchy between sections could be clearer.
- **Missing:** Specific memory budget calculations for the new model on ESP32-C3, concrete test fixture examples for v2 manifest rendering.

## 4. Architectural Philosophy

### 4.1 Core tenets

These principles guide every decision in this plan:

**Manifest-first.** `config/sensors.json` (v2) is the single source of truth. Everything — C++ structs, YAML fragments, dashboard defaults, test fixtures, API responses — is generated from it. No runtime JSON parsing on the ESP32.

**WHY:** This is the pattern already established in v7.4.5.0 and it works. Configuration drift between files was explicitly identified as a bug source (BUG-028). The generator model eliminates that class of bugs while keeping runtime behavior deterministic.

**Metadata-aware, not data-aware.** The firmware and dashboard do not know what "temperature" means. They know that a sensor has metrics with IDs, types, units, and rendering hints. The only place that "temperature" has semantic meaning is in the adapter that parses ThermoPro BLE packets.

**WHY:** This is what makes the system extensible. A ping probe and a ThermoPro sensor are both "a device with metrics" from the dashboard's perspective. The difference is in the adapter (BLE parser vs ICMP probe) and the manifest metadata (metric type, unit, display hints), not in the core data model.

**Bounded and predictable.** No `std::vector`, no `malloc` for core sensor state. Fixed-size arrays sized at compile time by the generator. Maximum metrics per sensor is capped. Flash segment sizes are deterministic.

**WHY:** The ESP32-C3 has ~320 KB of usable RAM and a single core. Heap fragmentation from dynamic allocation can cause hard-to-debug failures hours or days after boot. The current persistence engine works because segment sizes are mathematically predictable. That property must be preserved.

**Satellite simplicity, aggregation above.** Satellites remain self-contained edge appliances with simple, stable APIs. Aggregation is a separate role that runs on more capable hardware. The aggregator is a consumer of the satellite API, not an extension of the satellite firmware.

**WHY:** Trying to make a C3 be both a BLE scanner and a multi-gateway aggregator adds complexity, memory pressure, and failure modes. The current product identity — "open a browser to the ESP's IP and see your sensors" — is preserved for satellites. The aggregator is an additive capability, not a mutation.

**Validate with a real second use case.** Design the abstraction for generality, but do not finalize it until a real non-ThermoPro sensor family (ping probe, host metrics) is running end-to-end.

**WHY:** The repo's own roadmap says this: "Don't build a generic framework speculatively." Abstractions designed without concrete second use cases tend to be wrong in ways that only become visible when the second use case arrives. The ping probe is a zero-hardware-cost way to pressure-test the abstraction.

### 4.2 Conceptual model

The system is organized around three layers:

**Layer 1 — Source Adapters.** Where data comes from. Each adapter knows how to parse its specific input format (BLE packet, ICMP reply, HTTP response, GPIO read) and emit metric updates.

Examples: `thermopro_ble`, `icmp_ping`, `http_probe`, `local_system`, future `gpio_i2c`, `spi_sensor`

**Layer 2 — Logical Devices.** The things shown on the dashboard. Each device has an identity, a category, and a catalog of measurements. Devices are what users name and organize.

Examples: "Office Climate", "WAN Latency", "NAS Health", "Garage Leak Detector"

**Layer 3 — Measurements.** The actual values. Each measurement has a key, unit, type class (analog, binary, counter, etc.), and policies for history, display, and persistence.

Examples: `temp_c` (°C, analog, history enabled), `ping_ms` (ms, analog, RAM-only history), `leak_detected` (boolean, binary state, event history)

The manifest describes Layers 2 and 3. Layer 1 is implemented in firmware adapters. The dashboard renders Layers 2 and 3 from manifest metadata.

### 4.3 Measurement classes

Rather than making every metric fully custom, use a small taxonomy of measurement classes. This provides structure for rendering, charting, and persistence without requiring per-metric custom code.

**Analog numeric** — continuous float values like temperature, humidity, pressure, latency, CPU usage, disk usage. Line-chart friendly. Rolling average makes sense. Min/max tracking makes sense. This is the most common class.

**Binary state** — true/false values like leak detected, host reachable, API pass/fail. Badge/indicator display. Event history (state transitions) is more useful than continuous history.

**Counter / event-derived** — integer values like error count, reboot count, packet loss count. Summary display (total, rate) is more useful than dense charting.

**Metadata / support** — values like battery percentage, RSSI, last-seen timestamp, firmware version. Shown in card diagnostics and device health, not typically charted or persisted to flash.

**WHY this taxonomy:** It covers the full range of sensor types you described (environmental, acoustic, distance, host metrics, API checks) without requiring per-metric rendering logic. The dashboard can have one card renderer per device category that knows how to layout analog metrics, binary states, and counters using these classes. Adding a new metric of an existing class should require no new rendering code.

## 5. Manifest v2 — The Central Contract

### 5.1 Design requirements

The v2 manifest must:

- Describe gateway identity, role, and capabilities
- Describe each device with category, adapter, and source binding
- Describe each measurement with key, unit, type class, and display/history policies
- Be generated at compile time and served via HTTP endpoint
- Support backward compatibility with v1 (id/name/mac only)
- Be the single source consumed by: local dashboard, aggregator, CLI tools, and test fixtures

### 5.2 Recommended schema

```json
{
  "schema_version": 2,
  "gateway": {
    "id": "gw-bothell-01",
    "name": "Bothell Main Gateway",
    "role": "satellite",
    "hardware": "esp32-c3",
    "firmware_version": "7.6.0",
    "api_version": "v2"
  },
  "history": {
    "backend": "local_flash",
    "retention_hours": 1080,
    "ram_window_hours": 24,
    "sample_interval_seconds": 900
  },
  "devices": [
    {
      "id": "office",
      "name": "Office",
      "category": "environmental",
      "adapter": "thermopro_ble",
      "source": {
        "mac": "DB:06:2C:58:8A:59"
      },
      "measurements": [
        {
          "key": "temp_c",
          "label": "Temperature",
          "unit": "°C",
          "class": "analog",
          "data_type": "float",
          "history": true,
          "history_backend": "local_flash",
          "display": { "card": true, "chart": true, "summary": true }
        },
        {
          "key": "humidity_pct",
          "label": "Humidity",
          "unit": "%",
          "class": "analog",
          "data_type": "float",
          "history": true,
          "history_backend": "local_flash",
          "display": { "card": true, "chart": true, "summary": true }
        },
        {
          "key": "battery_pct",
          "label": "Battery",
          "unit": "%",
          "class": "metadata",
          "data_type": "float",
          "history": false,
          "display": { "card": true, "chart": false, "summary": false }
        },
        {
          "key": "rssi_dbm",
          "label": "RSSI",
          "unit": "dBm",
          "class": "metadata",
          "data_type": "float",
          "history": false,
          "display": { "card": true, "chart": false, "summary": false }
        }
      ]
    }
  ]
}
```

**WHY "devices" instead of "sensors":** The word "sensor" conflates the physical hardware (ThermoPro TP357), the logical entity (Office Climate), and the data it produces (temperature reading). "Device" cleanly refers to the logical entity, and "measurements" to its data. This avoids confusion as the system grows to include non-physical "devices" like ping probes or API health checks.

**WHY `class` instead of a rich type enum:** Measurement classes (analog, binary, counter, metadata) drive rendering and storage decisions. Specific types (temperature, humidity, ping_latency) are metadata for labeling but do not change how the system processes the data. Keeping the class vocabulary small means the dashboard and firmware only need a few code paths.

**WHY `display` hints:** These tell the dashboard whether to show a measurement in the card, chart, or summary sections. Battery and RSSI do not need chart history — encoding that in the manifest prevents the dashboard from having to know "battery means no chart." The manifest drives rendering, not hardcoded type checks.

**WHY `history_backend` per measurement:** When future cloud uploads or aggregator caching exist, some measurements might be "local_flash", some "ram_only", and some "cloud". Encoding this per-measurement lets the dashboard know where to fetch history from without probing multiple endpoints.

### 5.3 Backward compatibility

The v1 `config/sensors.json` format must continue to work during migration:

- `sensor_manifest_lib.py` gains a `to_manifest_v2()` function that takes a v1 manifest and fills in default category (`environmental`), adapter (`thermopro_ble`), and standard measurements (`temp_c`, `humidity_pct`, `battery_pct`, `rssi_dbm`) for each sensor.
- The generator (`render_sensor_config.py`) accepts either v1 or v2 input. If v1, it auto-promotes internally before rendering.
- The firmware serves both `/sensors.json` (v1 projection: `[{id, name}]`) and `/api/manifest` (v2 full). The v1 endpoint remains for any existing external consumers.

**WHY:** Existing installations, CLI scripts, and test fixtures should not break during the transition. The v1 → v2 auto-promotion means an operator can upgrade firmware without immediately rewriting their config file.

### 5.4 Compile-time generation

The Python generator reads `config/sensors.json` (v1 or v2) and produces:

- `dashboard/sensor_history_multi.h` — `SensorEntity` array, `MetricDef` arrays, compile-time constants
- `firmware/esp32-c3-multi-sensor.yaml` — BLE tracker blocks, sensor platform blocks, lambda references
- `dashboard/dashboard.js` — `DEFAULT_SENSOR_META` with v2 fields for offline/fallback rendering
- `dashboard/gateway_manifest.h` — C string literal containing the JSON v2 manifest, served by `/api/manifest`
- `tests/fixtures/sensors.json` — v2 manifest for Playwright tests

**WHY generated C string for `/api/manifest`:** The alternative (building JSON at runtime from C++ structs) requires a JSON serializer on the ESP32, adds heap pressure, and introduces failure modes. A compile-time string literal is zero-overhead and guaranteed correct.

## 6. Firmware Model — From SensorSlot to SensorEntity

### 6.1 The problem with SensorSlot

The current `SensorSlot` struct is a monolith that mixes identity, accumulation, formatting, and history:

```cpp
struct SensorSlot {
  const char* id;
  const char* name;
  const char* mac;
  float temp_sum, hum_sum;
  int temp_count, hum_count;
  float batt_last;
  uint32_t last_seen_epoch;
  HistoryBuffer temp_history, hum_history;
  char temp_avg_str[32], hum_avg_str[16], batt_str[16];
  bool temp_valid, hum_valid;
  void add_temp(float), add_hum(float), compute_and_format(uint32_t);
};
```

This cannot represent a device with CPU/RAM/disk metrics, a ping probe with latency/success values, or a leak sensor with a binary state. Every new measurement type would require adding fields and methods to SensorSlot, which is the definition of "hardcoded."

### 6.2 Proposed model

Three new types replace SensorSlot:

```cpp
// Generated at compile time by render_sensor_config.py

#define MAX_METRICS_PER_DEVICE 4
// WHY 4: covers ThermoPro (temp + hum + battery + rssi),
// host health (cpu + ram + disk + uptime), network (ping + success + uptime + loss).
// Keeps memory bounded and predictable.

struct MetricDef {
  const char* key;        // "temp_c", "humidity_pct", "ping_ms"
  const char* label;      // "Temperature", "Humidity"
  const char* unit;       // "°C", "%", "ms"
  uint8_t class_id;       // 0=analog, 1=binary, 2=counter, 3=metadata
  bool history_enabled;   // whether this metric has a history buffer
};

struct MetricState {
  float current_value;    // latest value or NAN
  float accumulator;      // for rolling average
  int sample_count;       // samples since last average
  bool valid;             // whether current_value is trustworthy
  uint32_t last_update_epoch;
  HistoryBuffer* history; // nullptr if history_enabled == false
};

struct SensorEntity {
  // Identity (from manifest)
  const char* id;
  const char* name;
  uint8_t category_id;       // 0=environmental, 1=system, 2=network, ...
  const char* adapter;       // "thermopro_ble", "icmp_ping", ...
  
  // Metrics (generated static arrays)
  const MetricDef* metric_defs;
  MetricState metric_states[MAX_METRICS_PER_DEVICE];
  uint8_t metric_count;      // actual metrics for this device (≤ MAX)
  
  // Adapter-specific fields
  const char* mac;            // non-null only for BLE devices
  int8_t last_rssi;
  uint32_t last_seen_epoch;
  
  // Generic methods
  void add_sample(uint8_t metric_index, float value);
  void compute_averages(uint32_t epoch);
  void mark_seen(uint32_t epoch);
};
```

**WHY `MAX_METRICS_PER_DEVICE = 4` as a hard cap:** A bounded maximum means every `SensorEntity` is the same size in memory. The compiler can compute `sizeof(SensorEntity)` at compile time. NVS segment allocation becomes predictable. The alternative — variable-length metric arrays — requires dynamic allocation and makes persistence layout dependent on per-device metric counts.

**WHY `HistoryBuffer*` pointer instead of inline buffer:** Not all metrics need history. Battery and RSSI are metadata-class metrics that only need a current value. Allocating a 96-entry ring buffer for a metric that never charts it would waste ~768 bytes per unused metric. The pointer is set by the generator: `nullptr` for non-history metrics, points to a statically allocated buffer for history metrics.

**WHY keep `mac` and `last_rssi` on SensorEntity:** These are adapter-specific fields that only matter for BLE devices. The alternative is a separate per-adapter struct, but that adds indirection and complexity for a small amount of data. Having them on the entity with `nullptr` for non-BLE devices is simpler and has negligible memory cost.

### 6.3 Generated code example

For a gateway with one ThermoPro sensor and one ping probe, the generator produces:

```cpp
// ── MetricDef arrays (const, flash-resident) ───────────────
static const MetricDef metrics_office[] = {
  {"temp_c",       "Temperature", "°C", 0, true},
  {"humidity_pct", "Humidity",    "%",  0, true},
  {"battery_pct",  "Battery",     "%",  3, false},
  {"rssi_dbm",     "RSSI",        "dBm",3, false}
};

static const MetricDef metrics_wan_ping[] = {
  {"ping_ms",      "Latency",     "ms", 0, true},
  {"success_pct",  "Success",     "%",  0, true}
};

// ── History buffers (only for history-enabled metrics) ──────
static HistoryBuffer hbuf_office_temp;
static HistoryBuffer hbuf_office_hum;
static HistoryBuffer hbuf_wan_ping_latency;
static HistoryBuffer hbuf_wan_ping_success;

// ── Device array ────────────────────────────────────────────
static constexpr int NUM_DEVICES = 2;

static SensorEntity devices[NUM_DEVICES] = {
  {
    .id = "office", .name = "Office",
    .category_id = 0, .adapter = "thermopro_ble",
    .metric_defs = metrics_office,
    .metric_states = {
      {.history = &hbuf_office_temp},
      {.history = &hbuf_office_hum},
      {.history = nullptr},
      {.history = nullptr}
    },
    .metric_count = 4,
    .mac = "DB:06:2C:58:8A:59"
  },
  {
    .id = "wan_ping", .name = "WAN Latency",
    .category_id = 2, .adapter = "icmp_ping",
    .metric_defs = metrics_wan_ping,
    .metric_states = {
      {.history = &hbuf_wan_ping_latency},
      {.history = &hbuf_wan_ping_success},
      {}, {}
    },
    .metric_count = 2,
    .mac = nullptr
  }
};
```

**WHY generated, not parsed:** This approach produces zero-overhead static data. No JSON parsing at runtime, no heap allocation for strings, no failure modes on boot. The Python generator validates the manifest at build time and produces C++ that the compiler verifies. If the manifest is malformed, you get a build error, not a runtime crash.

### 6.4 Adapter integration

Each adapter type needs a thin integration layer. For ThermoPro BLE (the existing path), this means modifying the YAML lambda that currently calls `sensors[i].add_temp(value)` to instead call `devices[i].add_sample(0, value)` where 0 is the index of `temp_c` in that device's metric array.

For a new ping probe adapter, a periodic task calls `devices[1].add_sample(0, latency_ms)` and `devices[1].add_sample(1, success_pct)`.

**WHY index-based, not key-based:** Looking up metrics by string key at runtime on every sample would add string comparison overhead to a hot path. The generator assigns metric indices at compile time, so the adapter code uses constant indices. This is the same approach used by the current `add_temp()` / `add_hum()` calls, just generalized.

## 7. Dashboard — Renderer Registry and Dynamic Cards

### 7.1 Current state

The dashboard is already more generic than the firmware. It loads a manifest, renders cards dynamically, and has a plugin system. But `buildSensorCards()` generates hardcoded temperature/humidity/dew point/comfort HTML, and `fetchSensorHistoryRows()` fetches exactly `/history/{id}/temp` and `/history/{id}/hum`.

### 7.2 Card renderer registry

Replace the monolithic `buildSensorCards()` with a category-dispatched system:

```javascript
var CARD_RENDERERS = {
  environmental: function(device) {
    // Current ThermoPro card layout: temp, avg temp, hum, avg hum,
    // dew point, comfort, min/max, battery, RSSI
    // This is the existing buildSensorCards() logic, scoped to one device.
    return buildEnvironmentalCard(device);
  },
  network: function(device) {
    // Ping latency value, success rate, sparkline if history available
    return buildNetworkCard(device);
  },
  system: function(device) {
    // CPU gauge, RAM bar, disk bar
    return buildSystemCard(device);
  }
};

// Main render loop
function buildDeviceCards() {
  var grid = document.getElementById('sensorGrid');
  grid.innerHTML = '';
  DEVICES.forEach(function(device) {
    var renderer = CARD_RENDERERS[device.category];
    if (!renderer) renderer = CARD_RENDERERS._default;
    grid.innerHTML += renderer(device);
  });
}
```

**WHY category-based, not fully generic per-metric:** A fully generic system that dynamically creates DOM elements for every metric sounds elegant but produces poor UX. A temperature + humidity card has dew point, comfort estimate, and min/max — concepts that do not apply to a CPU usage metric. Category-specific renderers let you design good UX per device type while still being manifest-driven. The renderer looks at `device.measurements` to know what to render, but the layout is designed for that category.

**WHY `_default` fallback:** When someone adds a new device category before a renderer exists, the dashboard should not crash. A default renderer that shows device name + raw metric values is better than a blank card. This matches the existing project philosophy of graceful degradation.

### 7.3 Measurement formatters

Alongside card renderers, add measurement-level formatting:

```javascript
var METRIC_FORMATTERS = {
  temperature: function(value, unit) {
    if (unit === '°C' || unit === 'C') {
      var f = value * 9/5 + 32;
      return value.toFixed(1) + ' °C / ' + f.toFixed(1) + ' °F';
    }
    return value.toFixed(1) + ' ' + unit;
  },
  humidity: function(value) { return value.toFixed(1) + ' %'; },
  ping_latency: function(value) { return value.toFixed(0) + ' ms'; },
  cpu_usage: function(value) { return value.toFixed(1) + ' %'; },
  _default: function(value, unit) { return value.toFixed(1) + ' ' + (unit || ''); }
};
```

**WHY:** Formatting logic is currently scattered across `compute_and_format()` in C++ and various DOM update functions in JS. A formatter registry centralizes it and makes it extensible. Adding a new metric type means adding one formatter entry, not hunting through multiple functions.

### 7.4 Generic history fetching

Replace the hardcoded temp/hum fetch:

```javascript
// New generic approach
function fetchDeviceHistory(device) {
  var historyMetrics = device.measurements.filter(function(m) {
    return m.history && m.display.chart;
  });
  
  var fetches = historyMetrics.map(function(metric) {
    return fetch(ESP_HOST + '/api/v2/history/' + device.id + '/' + metric.key,
                 {cache: 'no-store'})
      .then(function(r) { if (!r.ok) throw new Error('HTTP ' + r.status); return r.text(); })
      .then(function(text) { return parseHistoryCsv(text, metric.key); });
  });
  
  return Promise.all(fetches);
}
```

**WHY:** The current approach fetches exactly two series (temp, hum) per sensor. A device with CPU/RAM/disk has three chartable metrics. A ping probe has two. The fetch logic must be driven by the manifest's measurement definitions, not by hardcoded assumptions.

### 7.5 Boot flow change

```
Current:
  1. fetch /sensors.json → [{id, name}]
  2. hardcoded: each sensor has temp, hum, battery, rssi
  3. build ThermoPro cards

New:
  1. fetch /api/manifest → v2 manifest
     (fallback: fetch /sensors.json → auto-promote to v2 with ThermoPro defaults)
  2. build SensorEntity[] from manifest.devices
  3. for each device, dispatch to CARD_RENDERERS[device.category]
  4. fetch history per device's chartable measurements
  5. render charts using manifest metadata for labels, units, colors
```

**WHY the fallback:** During migration, some satellites may still be running older firmware without `/api/manifest`. The fallback ensures the dashboard works with both old and new firmware.

## 8. Gateway Aggregation Architecture

### 8.1 Roles

**Satellite (role = "satellite"):**
- Runs BLE scanning, local sensor adapters, local dashboard
- Exposes: `/api/status`, `/api/manifest`, `/api/v2/live`, `/api/v2/history/{device}/{metric}`, `/api/storage-stats`
- Does not know about other gateways
- Runs on ESP32-C3 (current hardware)
- Is independently useful — a user can open its dashboard and see their sensors

**Aggregator (role = "aggregator"):**
- Polls satellite APIs on a configurable schedule
- Caches satellite manifests and latest values in RAM
- Serves a unified dashboard showing all gateways and their devices
- Optionally has its own local sensors (if the hardware supports it)
- Recommended hardware: ESP32-S3 (more RAM, more sockets) or a Raspberry Pi / lightweight server for larger deployments

### 8.2 Aggregator configuration

A separate `config/aggregator.json`:

```json
{
  "schema_version": 1,
  "satellites": [
    {
      "id": "gw-bothell-01",
      "name": "Bothell Main",
      "base_url": "http://192.168.10.20",
      "poll_interval_seconds": 30,
      "auth": null
    },
    {
      "id": "gw-garage-01",
      "name": "Garage Sensors",
      "base_url": "http://192.168.10.21",
      "poll_interval_seconds": 30,
      "auth": { "type": "basic", "username": "admin", "password_ref": "secrets.garage_pw" }
    }
  ]
}
```

**WHY a separate file:** Satellite configuration (`config/sensors.json`) describes what sensors a gateway has. Aggregator configuration describes what other gateways to poll. Mixing them would conflate two different concerns. A satellite does not need to know about other gateways.

**WHY `password_ref` instead of inline password:** Passwords should not be committed to the repo. `password_ref` points to a key in the secrets file, matching the existing ESPHome secrets pattern.

### 8.3 Aggregator polling model

The aggregator runs a background task that, for each satellite:

1. Fetches `GET /api/manifest` — cached aggressively, refreshed every 5 minutes or on error
2. Fetches `GET /api/status` — every `poll_interval_seconds` (default 30s)
3. Fetches `GET /api/v2/live` — every `poll_interval_seconds`, stores latest values in RAM
4. Does NOT fetch history proactively — history is fetched on-demand when a user opens a chart for a specific device

**WHY on-demand history:** History responses can be large (45 days × hourly points × N metrics). Polling history for all satellites continuously would consume too much RAM and bandwidth. The aggregator dashboard fetches history directly from the satellite when a user drills into a chart — the aggregator proxies or redirects the request.

**WHY not browser-driven pull with CORS:** Three reasons:

1. **Network topology.** In any deployment beyond a flat home LAN — VLANs, VPNs, corporate firewalls — the user's browser cannot reach all satellites directly. The aggregator can, because it is on the same network segment.

2. **Security surface.** Setting `Access-Control-Allow-Origin: *` on every satellite exposes their APIs to any origin. An aggregator that proxies requests controls the access surface.

3. **Single integration point.** External tools (Grafana, Prometheus, monitoring scripts) should call one API endpoint, not discover and iterate over multiple satellite IPs.

### 8.4 Aggregator dashboard

The aggregator dashboard extends the existing dashboard with:

- **Gateway selector** — tabs or dropdown to choose which satellite's data to view, plus an "All Gateways" summary view
- **Gateway health cards** — shows each satellite's status (online/offline/stale), firmware version, uptime, free heap
- **Namespaced device IDs** — devices are prefixed with gateway ID to avoid collisions (e.g., `gw-bothell-01.office` and `gw-garage-01.office`)
- **Stale/unreachable indicators** — if a satellite stops responding, its devices are grayed out with "Last seen: X minutes ago"
- **History proxy** — when a user opens a chart, the dashboard requests history through the aggregator, which fetches it from the appropriate satellite

**WHY a shared dashboard codebase:** The satellite dashboard and aggregator dashboard share the same card renderers, metric formatters, and chart builders. The aggregator adds a gateway dimension but reuses the per-device rendering. This avoids maintaining two separate dashboards.

### 8.5 What the satellite must expose for aggregation

The existing endpoints already cover most of the aggregator's needs:

| Endpoint | Purpose | Exists today? |
|---|---|---|
| `GET /api/status` | Gateway health, uptime, heap, sensor count | Yes |
| `GET /api/manifest` | v2 manifest with gateway metadata, devices, measurements | New (Phase 1) |
| `GET /api/v2/live` | Current values for all devices/metrics | New (Phase 3) |
| `GET /api/v2/history/{device}/{metric}` | History stream for one measurement | New (Phase 3) |
| `GET /api/storage-stats` | Flash/RAM history bounds and retention | Yes |
| `GET /sensors.json` | Legacy v1 sensor list | Yes (kept for backward compat) |
| `GET /history/{id}/temp` and `/hum` | Legacy history endpoints | Yes (kept for backward compat) |

**WHY not add a single `/api/v2/everything` endpoint:** Because different consumers need different data at different frequencies. The manifest changes rarely. Status changes every few seconds. Live values change every polling cycle. History is expensive and on-demand. Splitting these lets consumers (and the aggregator) fetch only what they need at appropriate intervals.

## 9. API Contract Design

### 9.1 Existing endpoints (preserved)

All current endpoints remain unchanged for backward compatibility:

- `/sensors.json` — v1 sensor list
- `/history/{id}/temp`, `/history/{id}/hum` — temp/hum history
- `/api/status` — gateway health
- `/api/storage-stats` — flash/RAM usage and retention
- `/api/import/*` — CSV import
- `/api/reboot`, `/api/delete-data` — management actions

### 9.2 New v2 endpoints

**`GET /api/manifest`** — Returns the v2 manifest JSON (gateway metadata + devices + measurements). This is the "contract of record" that the dashboard, aggregator, and external tools consume.

**`GET /api/v2/live`** — Returns current values for all devices and metrics:

```json
{
  "timestamp": 1710264000,
  "devices": {
    "office": {
      "temp_c": 23.4,
      "humidity_pct": 45.2,
      "battery_pct": 87,
      "rssi_dbm": -62,
      "last_seen": 1710263985
    },
    "wan_ping": {
      "ping_ms": 12.3,
      "success_pct": 100,
      "last_seen": 1710263990
    }
  }
}
```

**WHY flat key-value per device:** The aggregator and dashboard poll this frequently (every 15–30 seconds). A flat map is fast to serialize, small to transfer, and simple to consume. The manifest already describes what each key means.

**`GET /api/v2/history/{device_id}/{metric_key}`** — Returns history in `epoch,value\n` format (same format as existing `/history/{id}/temp`). This means existing chart parsing logic works unchanged — only the URL changes.

**WHY keep the same CSV-line format:** The existing `parseHistoryCsv()` in the dashboard already handles this format. Changing to JSON would be a breaking change to the chart pipeline for no benefit.

**`POST /api/ingest/{device_id}/{metric_key}?val={float}`** — Allows external systems to push data into the gateway. A script on a NAS can periodically call `POST /api/ingest/nas01/cpu_usage_pct?val=33.2` to send system metrics to the gateway.

**WHY:** Some metrics originate off-device. The gateway should not need to implement an HTTP client for every remote data source. A simple push endpoint lets external scripts push data in, keeping the gateway firmware simple.

## 10. History and Persistence Strategy

### 10.1 Short-term: keep ThermoPro flash unchanged

The current `SegmentSnapshot` structure stores `temp[NUM_SENSORS][POINTS]` and `hum[NUM_SENSORS][POINTS]`. This layout is specific to ThermoPro but it works, it has been tested, and it supports 45-day retention with known flash wear characteristics.

**For v7.5–v7.6, do not change this.** Environmental metrics (ThermoPro temp/hum) continue to use the existing flash persistence exactly as today.

**WHY:** The persistence engine is the most delicate part of the codebase. It has specific segment size assumptions, NVS key naming, boot-time restoration, and import/merge logic. Changing it to accommodate variable metric counts would require rewriting the SegmentSnapshot format, the NVS key scheme, the boot restore path, the CSV export/import format, and all associated tests. That is a high-risk change with limited immediate value.

### 10.2 New metric categories: RAM-only initially

Non-environmental metrics (network, system) use RAM ring buffers only — the same `HistoryBuffer` class that already exists, but without flash persistence. This gives you:

- "Last 24 hours" of history for new metric types (same as RAM history today)
- Zero flash format changes
- Zero migration risk

**WHY this is acceptable for new categories:** Ping latency and CPU usage are operational metrics where recent history (hours, not months) is most valuable. The business use case for 45-day ping history is weak compared to 45-day temperature history. When you do need long-term retention for non-climate metrics, that can be addressed in v8.x with a generalized flash format or external storage.

### 10.3 Long-term: generalized flash persistence (v8.x)

When the metric model has been validated with real deployments and the need for long-term non-climate history is concrete, design a new flash format keyed by `(device_id, metric_key, epoch)` with a version tag and documented migration path. This should be a separate design effort informed by actual usage patterns.

### v7.5.7.0 — Aggregator Manifest Truncation Fix + PSRAM-Aware Scaling

**Status:** COMPLETE (v7.5.7.0, 2026-03-28)

**What changed:**

- Increased aggregator manifest cache and fetch temp buffers from 4096 to 8192 via `AGG_MANIFEST_BUF_SIZE`.
- Added a truncation guard in `/api/aggregator/gateways`: when cached manifest length hits the truncation threshold (`>= AGG_MANIFEST_BUF_SIZE - 1`), emit `"manifest":null` and log a warning instead of embedding potentially invalid JSON.
- Updated generator behavior to enforce PSRAM-aware aggregator enablement:
  - No PSRAM + `aggregator.json` present → generate `AGGREGATOR_ENABLED 0` (satellite-only) with warning.
  - PSRAM present → allow aggregator and cap `MAX_SATELLITES` at 8.
  - Generated satellite arrays are sliced to the capped satellite count.

**Buffer increase rationale:**

Satellite manifests are variable-length and grow as devices/metrics/categories are added. A 4096-byte cap could silently truncate a valid manifest, and embedding that truncated JSON inside the gateways payload breaks the full response. Raising to 8192 plus explicit truncation detection preserves payload validity under growth.

**PSRAM-aware scaling decision:**

Aggregator role remains constrained to PSRAM-equipped boards to keep memory safety predictable in RAM-constrained environments. Enforcing this rule in `render_sensor_config.py` allows one shared repository/config flow while ensuring non-PSRAM boards stay in safe satellite mode.

**Relationship to Phase D (v7.6.0.x):**

This step lays groundwork for runtime satellite management by hardening buffer infrastructure and compile-time scaling constraints before introducing dynamic satellite lifecycle endpoints. Phase D can now build on safer manifest handling and explicit board capability gating.

## 13. Business Readiness Considerations

### 13.1 What makes someone say "I want that"

The product story is: "Plug in a $5 ESP32 board, configure your sensors in one JSON file, flash it, and open a browser to see your data. No cloud, no subscriptions, no dependencies. Add more gateways, add different sensor types, and see everything in one place."

That pitch works because:

- The barrier to entry is low (one JSON file, one flash)
- The dependency count is zero (no Home Assistant, no database, no cloud account)
- The data stays local (compelling for businesses, privacy-conscious users, air-gapped environments)
- The dashboard is immediately useful (not a "connect to your existing Grafana" story)

### 13.2 What needs to happen for business readiness

**API versioning** — Already addressed in this plan (`/api/v2/` namespace). Critical for not breaking existing integrations when the API evolves.

**Provisioning workflow** — Currently requires editing JSON + running Python scripts + flashing. For a business product, the goal should be: "run one setup command, answer prompts, flash." The manifest + generator pattern is the foundation for this. A future CLI wizard could be built on top.

**Observability** — The `/api/status` endpoint is a good start. Add: adapter error counts, sensor stale counts, persistence health (NVS free space, last write status), import/export status. This matters for support — when a customer says "it stopped working," you need diagnostic data.

**Upgrade safety** — The manifest + generator pattern and NVS version magic numbers already provide good guardrails. Document the upgrade path clearly: what firmware versions are compatible with what manifest versions.

**Documentation** — The existing docs are good for a developer. For a business product, add: a "Quick Start" guide (5 steps, no background knowledge required), a "Troubleshooting" page, and a "Supported Sensor Types" page with adapter-specific setup instructions.

### 13.3 Hardware recommendations

**Satellites:** ESP32-C3 SuperMini. Current choice is correct — low cost, low power, sufficient for BLE + WiFi + local dashboard + 4 sensors. Keep this as the standard satellite platform.

**Aggregator (small deployments, up to ~5 satellites):** ESP32-S3. More RAM (~512 KB), dual-core, better suited for concurrent HTTP clients and larger dashboard payload. Still cheap and embeddable.

**Aggregator (larger deployments, >5 satellites):** Raspberry Pi Zero 2W or similar. Runs a lightweight web server (Node.js, Python Flask, or static HTML + JavaScript) that polls satellites and serves the aggregator dashboard. This is the most capable and most cost-effective option for non-trivial deployments.

## 14. What Not to Do

These are guardrails based on the failure patterns identified across all feedback sources and the repo's own bugs-and-lessons-learned.

### Do not build a generic framework before having a second real sensor type

The repo's own roadmap says it correctly. Design the abstraction, but validate it with a concrete second use case before finalizing. Phase 4 (ping probe) exists for exactly this purpose.

### Do not dynamically resize NVS segments based on metric count

The current fixed-size `SegmentSnapshot` is a strength. Coupling manifest edits to mandatory history wipe + migrate + restore is a poor user experience and a data loss vector. Defer flash format generalization to v8.x.

### Do not use dynamic allocation for core sensor state

No `std::vector`, no `malloc`, no runtime-sized arrays for metrics or history buffers. The generator produces fixed-size static arrays. This is essential for ESP32 memory stability.

### Do not break existing endpoints

Add `/api/v2/` alongside existing endpoints. Do not rename or remove `/sensors.json`, `/history/{id}/temp`, or `/api/status`. External consumers and older dashboards depend on them.

### Do not force the aggregator into the ESP32-C3

The C3 is the right satellite. It is not the right aggregator for anything beyond a trivial deployment. Recommend ESP32-S3 or Raspberry Pi for aggregation.

### Do not tie dashboard rendering to adapter names

Render by device category and measurement class, not by "if adapter == thermopro_ble, show temperature card." The adapter is an implementation detail. The category and measurements drive the UX.

### Do not over-engineer the first aggregator

The aggregator MVP should do three things: poll satellites, cache manifests + latest values, serve a unified dashboard. It does not need to handle satellite firmware updates, manage satellite configuration remotely, or implement complex caching strategies. Keep it simple and add capabilities based on real usage feedback.

## Appendix C — Glossary

| Term | Meaning |
|---|---|
| **Device** | A logical entity on the dashboard (e.g., "Office Climate", "WAN Latency") |
| **Measurement** | A single data value produced by a device (e.g., `temp_c`, `ping_ms`) |
| **Adapter** | Source-specific logic that translates raw input into metric updates |
| **Category** | A classification of device type (environmental, network, system) |
| **Measurement class** | A classification of metric behavior (analog, binary, counter, metadata) |
| **Satellite** | A gateway that scans sensors and serves data locally |
| **Aggregator** | A gateway (or server) that polls satellites and presents a unified view |
| **Manifest v2** | The JSON contract describing a gateway's devices and measurements |
| **Card renderer** | A JavaScript function that generates HTML for one device card |
| **Metric formatter** | A JavaScript function that formats a metric value for display |
