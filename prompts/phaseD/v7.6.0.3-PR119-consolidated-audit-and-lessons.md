## Part II — Did the Codebase State Match the Prompt's Assumptions?

**Mostly, but with four documented mismatches.**

### Assumptions that matched

1. `probe_satellite_manifest_()` existed, was unmodified since v7.6.0.1, and was directly reusable
2. `POST /api/aggregator/test-satellite` was still wired to the 501 stub and ready to replace
3. The step was correctly scoped as a read-only probe with no NVS side effects
4. Mechanical regeneration/version churn was expected

### M1 — Auth behavior from the replaced stub was omitted from the prompt snippet

The replaced `handle_aggregator_stub_501_()` called `authenticate_management_(request)` as its
first action. The prompt §5d code block omitted this call. The agent faithfully reproduced the
omission. The fixup added it.

**Assessment:** real prompt defect / real codebase-state mismatch.

### M2 — Prompt parsing patterns assumed compact JSON; manifests are pretty-printed

The prompt's `hardware` extraction used `strstr(s_proxy_tmp, "\"hardware\":\"")` (no space after
colon). The existing `probe_satellite_manifest_()` helper in the same file already uses a
whitespace-tolerant skip pattern for `id` and `name`. The project's manifest fixtures confirm
pretty-printed output (`"hardware": "ESP32-C3"`). The fixup corrected both parsers.

**Assessment:** real prompt defect / real codebase-state mismatch.

### M3 — Prompt contract omitted the inherited management POST non-empty-body branch

The endpoint was placed inside `is_management_post_route_()` which causes `handleRequest()` to
reject empty-body POST requests with `400 "Non-empty body required for management POST"` before
the handler is invoked. This branch was absent from the §5a / §12 contract tables, the
changelog, and the session log in the initial commit. The fixup updated the changelog and session
log to document it.

**Assessment:** real prompt defect / real contract-documentation mismatch.

### M4 — G3/G4 handoff notes describe an earlier prompt snapshot, not the current file on main

The handoff document lists G3 (`String` → `std::string`) and a version-typo bug as known prompt
issues. The prompt file currently on `main` already shows `std::string` and the correct bump
command. Any analysis of G3 must clarify it describes a prior prompt state, not the current
file.

**Assessment:** documentation-state mismatch within the audit source set. Not a code defect.

### Where the prompt aligned with the guide

The prompt told the agent to read `probe_satellite_manifest_()`, `send_json_error_()`,
`add_common_headers_()`, and the 501 route wiring before making any changes.

**Guide §3.4 — CI-exact pre-conditions.**
The prompt included Playwright, preflight, and render-check commands with exact invocations.

**Guide §3.5 — Exact scope with data-flow tracing.**

**Guide §3.6 — Do NOT section.**
The prompt had a crisp, file-specific scope boundary. It correctly excluded mechanical version

**Guide §3.13 — Complete code block for the handler (§5d).**
A copy-ready implementation was provided. This reduced ambiguity on data-flow but introduced
the prompt defects (P1, P2) because the snippet was not production-safe.

### Where the prompt violated the guide's intent

**G-Audit-1 — Prompt code block was not production-safe (§3.13).**
The §5d code snippet omitted the auth gate, used compact-JSON parsing, lacked truncation
handling, and did not note the inherited management POST body branch. A code block labeled
copy-ready must be held to production standards.

**G-Audit-2 — Contract fidelity was incomplete (§3.12 / §3.5).**
The §5a / §12 contract tables omitted the inherited `400 Non-empty body required for management
POST` branch. Contract tables must cover all branches the caller will observe, including those
applied by `handleRequest()` before the handler fires.

**G-Audit-3 — Historical prompt state not flagged as stale (§11 Prompt Maintenance).**
The handoff document preserved G3/G4 as "known prompt issues" without noting they had already
been corrected in the prompt file on `main`. Stale issue entries mislead future audits.

### Guide-driven improvements to carry forward

1. **Auth Policy Block** — Every endpoint prompt section must include:
   ```
   Auth: REQUIRED / NOT REQUIRED
   If REQUIRED: add `if (!authenticate_management_(request)) return;` after method gate
   If NOT REQUIRED: comment: // Auth intentionally omitted — [rationale]
   ```

2. **Inherited Route-Level Behavior block** — Every endpoint contract must document all branches
   applied by `handleRequest()` before the handler fires, including:
   - empty-body guard for management POST routes
   - transport-layer vs handler-layer method rejection (LESSON-OPS-109)

3. **Production-safe code blocks** — All prompt code snippets must include auth gates, buffer
   truncation guards, and parsing styles that match the actual runtime JSON format.

4. **Fallback field behavior rows** in §12 contract tables:
   - `hardware` absent → `"unknown"`
   - `sensor_count` absent → `0`

5. **Handoff G-note lifecycle** — When a known prompt issue is corrected, mark it ✅ RESOLVED
   in the handoff document rather than leaving it as an open finding.

## Part VII — Remaining Issues and Deferred Fixes

### R1 — JSON string values still not escaped

`probe_id`, `probe_name`, and `hw_str` are inserted into the JSON response body via `snprintf`
`"%s"` without escaping. A satellite name containing `"` or `\` would produce invalid JSON.

**Current mitigation:** Code comment documents the limitation. Satellite IDs/names follow an
alphanumeric-plus-hyphens naming convention so special characters are not expected under normal
operation.

**Status:** deferred / accepted as known limitation
**Action for v7.6.0.4 prompt:** Add to prompt: "escape string fields or add note if convention
guarantees safe characters."

### R2 — `s_proxy_tmp` access is documented but not mutex-guarded

The comment added in `b43340c` explains why the buffer is safe to read without a mutex
(single-threaded ESP-IDF httpd dispatch). This is a pre-existing architectural pattern, not
introduced by this PR. No concurrent access is possible in the current request model.

**Status:** accepted within current architecture; comment is the correct resolution
**Note:** If the httpd ever becomes multi-threaded, this pattern must be revisited globally.

### R3 — Manual `strstr`-based parsing remains structurally fragile

Both `hardware` and `sensor_count` parsers are now whitespace-tolerant but still use manual
pointer arithmetic. They are correct for the current manifest format but would not survive
manifest-format changes (e.g., nested objects, arrays, renamed keys).

**Status:** deferred / accepted within current project constraints
**Note:** The entire Phase D infrastructure uses this pattern consistently. A structured JSON
parser is a broader architectural improvement, not scoped to v7.6.0.3.

### R4 — Device testing not yet evidenced

The `scripts/device-test-v7.6.0.3.sh` script exists and is ready. No execution transcript was
captured in the PR or session log. This is the mandatory step before tagging `v7.6.0.3`.

**Status:** blocking for tag — must be completed
**Action:** See Section VIII below.

### R5 — T7 (DELETE method) result needs script reconciliation

The original audit stated T7 returns `HTTP 000` because `canHandle()` does not register
`/api/aggregator/test-satellite` for `HTTP_DELETE`, causing a transport-layer rejection rather
than the handler's JSON 405. The device-test script expectation must be verified against the
actual runtime behavior.

This is consistent with LESSON-OPS-109 and the v7.6.0.2 BUG-079 experience. No code fix
required — the endpoint is POST-only and DELETE is correctly rejected, just at the transport
layer rather than the application layer.

**Status:** script expectation should be verified against actual device output before using T7
as a hard proof point

## Part IX — New Lessons

### LESSON-OPS-110 — Prompt code snippets for endpoint handlers must include explicit auth policy

Omitting `authenticate_management_()` from a prompt code block is a prompt defect that coding
agents will faithfully reproduce. The replaced stub's auth call is the authoritative reference
— if the replacement prompt omits it, the agent will drop it. From v7.6.0.4 onward, every
endpoint handler code block in a prompt must include an explicit auth decision:
```
Auth: REQUIRED / NOT REQUIRED
Rationale: [one sentence]
```

### Prompt Writing Guide additions

1. **Auth Policy Block** — mandatory in every endpoint prompt section (format above)
2. **Inherited Route-Level Behavior** — contract tables must list all branches applied by
   `handleRequest()` before the handler fires
3. **Production-safe code blocks** — whitespace-tolerant JSON parsing, truncation guards, auth
   gates; prompt code snippets are held to production standards
4. **Fallback field defaults in §12** — missing manifest fields must have explicit default rows
5. **G-note lifecycle** — resolved prompt issues must be marked ✅ RESOLVED, not left open

### No new critical rules

All issues are addressed by existing Critical Rules (Rule 44 for `std::string`) and the new
prompt writing guide additions above. The auth omission pattern does not warrant a new Critical
Rule — it is a prompt-writing discipline issue.

## Part X — Security Convention Summary (Phase D Endpoints)

| Endpoint | Method | Auth | Rationale |
|---|---|---|---|
| `POST /api/aggregator/add-satellite` | POST | ❌ Not required | Intentional — discovery/add from dashboard without pre-auth |
| `DELETE /api/aggregator/satellite/{id}` | DELETE | ✅ Required | Destructive — removes satellite from config + NVS |
| `POST /api/aggregator/test-satellite` | POST | ✅ Required | Reveals network topology (satellite IDs, hardware, sensor counts) |
| `POST /api/aggregator/reset-satellites` | POST | ✅ Required | Destructive — clears all satellites |

**Notable:** `add-satellite` is intentionally unauthenticated and documents why. `test-satellite`
was unauthenticated in the initial commit (mirroring the prompt snippet) and auth was added
during review. Going forward, the pattern is: destructive or topology-revealing management
endpoints require auth; add/discovery endpoints may be open with documented rationale.
